# README Style Guide

## Purpose

Keep project-facing documentation concise, consistent, and actionable for both new users and contributors.

## Required README Sections

1. Project overview and value proposition.
2. Quick start (build, run, test).
3. Architecture snapshot with links to ADRs.
4. Data seed/versioning explanation.
5. Legal/privacy/trademark links.
6. Contribution and issue reporting links.

## Writing Rules

- Use plain language over vendor jargon where possible.
- Prefer short paragraphs and bullet lists.
- Include command examples in fenced code blocks.
- Link to deep docs instead of repeating long procedures.

## Terminology Consistency

- Use "3D Game Optimizer" as product name.
- Use "compatibility tier" for capability levels.
- Use "display profile" for monitor/projector presets.
- Use "tool automation" for external executable orchestration.
